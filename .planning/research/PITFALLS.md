# Pitfalls Research

**Domain:** Public-facing AI tool frontend (Next.js + FastAPI) with long-running jobs, iframe preview, no auth
**Researched:** 2026-03-21
**Confidence:** HIGH

## Critical Pitfalls

### Pitfall 1: Cloudflare Tunnel Buffers SSE Streams, Breaking Real-Time Progress

**What goes wrong:**
Server-Sent Events (SSE) sent from FastAPI through a Cloudflare tunnel never arrive in real-time. The user stares at a blank progress screen for 30-60 seconds, then all events dump at once when the connection closes. The redesign appears to "hang" and then instantly complete, destroying the UX that justifies the wait.

**Why it happens:**
Cloudflare Quick Tunnels buffer GET responses at the edge. SSE over GET -- the standard browser EventSource pattern -- is fully buffered until the connection closes. This is a documented, long-standing issue (cloudflared#1449, cloudflared#199). Named tunnels handle this better but still have edge cases with HTTP/3 protocol selection and ~100s timeout interruptions.

**How to avoid:**
1. Use a **named tunnel**, not a Quick Tunnel. Named tunnels stream SSE correctly.
2. Use **POST-based SSE** as a fallback. POST streams work even through Quick Tunnels. This means using `fetch()` with `ReadableStream` on the frontend instead of the native `EventSource` API (which only supports GET).
3. Set `protocol: http2` in the cloudflared config to avoid HTTP/3 buffering quirks.
4. Send periodic keepalive events (every 15s) to prevent proxy idle timeouts.
5. Implement `Last-Event-ID` reconnection so network hiccups don't lose progress state.

**Warning signs:**
- Progress bar works on `localhost:3000` -> `localhost:8080` but not through the tunnel
- Events arrive in batches instead of individually
- SSE connection drops after ~100 seconds with a 200 status

**Phase to address:**
Phase 1 (API + streaming). This is architectural -- if you build SSE with EventSource (GET) first and discover the tunnel buffers it later, you have to rewrite the entire streaming client. Validate streaming through the tunnel on day one.

### Pitfall 2: iframe srcdoc Inherits Parent Origin, Enabling XSS Escape

**What goes wrong:**
The redesigned HTML (generated by Claude from crawled websites) is displayed in an iframe using `srcdoc`. If the iframe has `sandbox="allow-scripts allow-same-origin"`, the sandboxed content is same-origin with the parent page and can access the parent DOM, steal cookies, redirect the user, or modify the contact form. Even without `allow-same-origin`, `allow-popups` combined with `allow-popups-to-escape-sandbox` lets script in the iframe open a new window with full privileges.

This is not theoretical. The CONCERNS.md already flags that LLM-generated HTML is served without sanitization. A crawled page could contain prompt injection that causes Claude to embed `<script>` tags, fetch() calls, or `javascript:` hrefs in the redesigned output.

**Why it happens:**
Developers add `allow-scripts` so the redesigned page renders correctly (animations, interactive elements), then add `allow-same-origin` because some CSS features or font loading requires it. This combination is equivalent to no sandbox at all.

**How to avoid:**
1. Use `sandbox="allow-scripts"` **without** `allow-same-origin`. This gives the iframe a unique opaque origin, isolating it from the parent.
2. Add the `csp` attribute on the iframe: `<iframe csp="default-src 'unsafe-inline' data:; script-src 'none'; connect-src 'none'"` -- this blocks scripts entirely while allowing inline CSS and data URIs for images.
3. Serve redesigned HTML from a **separate origin** (e.g., `preview.sastaspace.com` or a different port through a second tunnel route) so that even if sandbox fails, the content cannot access the main domain's cookies or storage.
4. Strip `<script>` tags and event handler attributes (`onclick`, `onerror`, etc.) from the LLM output server-side using a library like `bleach` or `nh3` before serving.
5. Set `Content-Security-Policy: sandbox` header on the HTML served to the iframe as a defense-in-depth layer.

**Warning signs:**
- Console errors about blocked cross-origin access (means sandbox IS working)
- No console errors when iframe content tries to access `parent.document` (means sandbox is NOT working)
- Redesigned HTML containing `<script>` tags in the output

**Phase to address:**
Phase 1 (result display). The iframe security model must be designed before the first redesign is shown to the public. Retrofitting sandboxing after launch means every previously generated HTML file is a potential XSS vector.

### Pitfall 3: No Rate Limiting on a 30-60s CPU+LLM Job Enables Trivial DoS

**What goes wrong:**
Without authentication or rate limiting, anyone can submit unlimited redesign requests. Each request spawns a Playwright browser instance (200-400MB RAM) and a 30-60 second Claude API call through the local gateway. Three concurrent requests from a single bot saturate the local Mac. Ten requests crash it. The CONCERNS.md already identifies "No rate limiting or concurrency control" as a missing critical feature.

**Why it happens:**
The project explicitly scopes out authentication ("No auth: completely open, no signup required"). Developers assume rate limiting requires auth, or plan to "add it later." But for an expensive operation (browser + LLM), rate limiting is not a nice-to-have -- it is the primary defense against resource exhaustion.

**How to avoid:**
1. **IP-based rate limiting** at the FastAPI level: 1 concurrent request per IP, max 3 requests per hour per IP. Use `slowapi` (built on `limits`) or a simple in-memory dict with TTL.
2. **Global concurrency limit**: Maximum 1-2 concurrent redesign jobs. Queue additional requests with a position indicator. Use `asyncio.Semaphore(1)` wrapping the redesign pipeline.
3. **Cloudflare WAF rules**: Rate limit the `/api/redesign` endpoint at the Cloudflare level (5 requests per minute per IP) as a first line of defense before traffic even reaches the tunnel.
4. **Request timeout**: Hard-kill any redesign job that exceeds 90 seconds. Prevents zombie Playwright instances.
5. **Cloudflare Turnstile** on the form submission: invisible CAPTCHA that blocks automated submissions without friction for real users. Free tier, already in the Cloudflare ecosystem.

**Warning signs:**
- Memory usage spikes on the host Mac
- Multiple redesign processes running simultaneously in `ps aux`
- Playwright browser instances not being cleaned up
- Claude API gateway becoming unresponsive

**Phase to address:**
Phase 1 (API endpoint). Rate limiting and concurrency control must ship with the `/redesign` endpoint, not after. An unprotected public endpoint with 30-60s compute jobs will be found and abused within hours of going live.

### Pitfall 4: SSE Connection Drops Mid-Job With No Recovery, User Loses 30s of Waiting

**What goes wrong:**
The user submits a URL, waits 25 seconds watching progress, then the SSE connection drops (mobile network switch, laptop sleep, Cloudflare tunnel hiccup, browser tab backgrounding). The redesign continues server-side but the user sees an error or stale "Processing..." state. They resubmit, spawning a duplicate job. The original result is orphaned.

**Why it happens:**
SSE connections are fragile. Browsers throttle background tabs. Mobile networks switch between WiFi and cellular. Cloudflare tunnels have a ~100s idle timeout. Most implementations only handle the happy path.

**How to avoid:**
1. **Job ID pattern**: Return a job ID immediately on submission. SSE streams progress for that job ID. If the connection drops, the client reconnects and resumes from `Last-Event-ID`.
2. **Polling fallback**: If SSE fails 2 times, fall back to polling `GET /api/redesign/{job_id}/status` every 3 seconds. Less elegant but bulletproof.
3. **Idempotent submissions**: Hash the input URL and check if a redesign is already in progress or recently completed. Return the existing job instead of spawning a duplicate.
4. **Result persistence**: Store the completed HTML result for at least 1 hour. The result page URL (`/redesign/{job_id}`) should work whether the user watched the stream or not.
5. **Browser `visibilitychange` handler**: When the tab becomes visible again, immediately check job status rather than relying on the SSE stream that may have been throttled.

**Warning signs:**
- Users submitting the same URL multiple times in logs
- Orphaned Playwright processes with no client waiting for results
- "Connection reset" errors in SSE stream logs

**Phase to address:**
Phase 1 (API + frontend streaming). The job ID + polling fallback pattern must be the foundation of the streaming architecture, not bolted on later.

### Pitfall 5: Contact Form Spam Floods Inbox Within Days of Launch

**What goes wrong:**
A public contact form with no authentication or CAPTCHA receives 50-500 spam submissions per day within a week of being indexed. The owner's inbox becomes unusable, legitimate leads are buried in spam, and if using a transactional email service (SendGrid), the account may be flagged for spam.

**Why it happens:**
Bots crawl the web for exposed forms and submit automatically. A new site with a simple POST endpoint is an easy target. "I'll add spam protection later" means it ships without protection and spam starts before the fix.

**How to avoid:**
1. **Cloudflare Turnstile** (primary): Invisible challenge, free, integrates with the existing Cloudflare setup. Server-side validation of the Turnstile token is required -- client-side only is bypassable.
2. **Honeypot field** (secondary): Add a hidden `website` or `company` field that is visually hidden via CSS (not `display:none` which smart bots detect). If filled, reject silently.
3. **Time-based check**: Reject submissions that arrive less than 3 seconds after page load. No human fills a 3-field form that fast.
4. **Server-side validation**: Email format validation, message minimum length (>20 chars), reject if message contains more than 2 URLs.
5. **Store locally first**: Write submissions to a local JSON file or SQLite database, then forward via email. This prevents spam from hitting the email service directly and allows batch review.

**Warning signs:**
- Sudden spike in contact form submissions
- Submissions with gibberish or identical messages
- Email service sending rate warnings

**Phase to address:**
Phase 2 (contact form). Turnstile + honeypot must ship with the contact form, not after. The form should never go live without at least Turnstile validation.

### Pitfall 6: Next.js-to-FastAPI Proxy Breaks Under Streaming and Large Payloads

**What goes wrong:**
Developers route Next.js API routes as a proxy to FastAPI to avoid CORS. But Next.js API routes (especially on Vercel, but also in self-hosted mode) have payload size limits (~4.5MB default) and timeout limits. SSE streams through Next.js API routes buffer unpredictably. The redesigned HTML (which can be large single-file HTML with inline images) exceeds the payload limit and silently truncates.

**Why it happens:**
Using Next.js API routes as a proxy seems clean architecturally (single origin, no CORS). But Next.js API routes are designed for lightweight request/response, not streaming proxies for long-running jobs.

**How to avoid:**
1. **Do not proxy through Next.js API routes for the redesign endpoint.** Let the browser talk directly to FastAPI for the `/api/redesign` SSE stream.
2. Configure **CORS on FastAPI** with explicit origin allowlist: `allow_origins=["https://sastaspace.com"]`. This is simpler and more reliable than proxying.
3. Expose **both Next.js and FastAPI** through the Cloudflare tunnel on different paths (e.g., `/` -> Next.js, `/api/` -> FastAPI) using cloudflared ingress rules.
4. If you must proxy lightweight requests (contact form), set `bodyParser: { sizeLimit: '1mb' }` explicitly on those routes.
5. Use `next.config.js` `rewrites` for simple GET endpoints only, never for streaming.

**Warning signs:**
- SSE events arriving in Next.js dev mode but not in production build
- "Body exceeded 4.5MB limit" errors in Next.js logs
- Streaming responses arriving all at once instead of incrementally

**Phase to address:**
Phase 1 (architecture). The decision of how the browser reaches FastAPI must be made before any API integration code is written. Changing from proxy to direct later means rewriting every fetch call.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Storing redesign results only in filesystem (`sites/` dir) | No database setup needed | No metadata queries, no cleanup, no expiry, hard to list/search | MVP only -- add SQLite metadata index in Phase 2 |
| In-memory rate limiting (dict/semaphore) | No Redis dependency | Lost on server restart, no shared state if multiple workers | Acceptable for single-worker local deployment |
| Hardcoded Turnstile site key in frontend | Quick to ship | Cannot rotate keys without redeployment | MVP only -- move to env var |
| Inline CSS/JS in redesigned HTML | Single-file output, simple serving | CSP cannot distinguish legitimate inline from injected | Acceptable if iframe sandbox is strict |
| Single uvicorn worker (no gunicorn) | Simple startup, matches CLI model | Cannot handle concurrent requests during redesign | Acceptable for local deployment with concurrency=1 |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Cloudflare Tunnel + SSE | Using Quick Tunnel for development, then wondering why SSE breaks | Use named tunnel from day one; test streaming through tunnel early |
| Cloudflare Tunnel + multiple services | Single tunnel, single origin | Use cloudflared ingress rules to route `/` to Next.js (port 3000) and `/api/` to FastAPI (port 8080) |
| Cloudflare Turnstile | Client-side widget only, no server verification | Always verify the Turnstile token server-side via `https://challenges.cloudflare.com/turnstile/v0/siteverify` |
| FastAPI + Playwright | Running Playwright in the request handler thread | Playwright is async; ensure the FastAPI redesign endpoint is `async def` and the event loop is not blocked by synchronous subprocess calls |
| Next.js + FastAPI CORS | Setting `allow_origins=["*"]` in FastAPI | Explicitly list the frontend origin; wildcard `*` blocks credentials and is a security risk |
| Claude Code API gateway | Assuming the gateway is always running | Check gateway health before starting a redesign; return a clear error if the gateway is down rather than hanging for 30s |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Concurrent Playwright instances | Mac fans spinning, 1GB+ RAM usage, OOM kills | `asyncio.Semaphore(1)` around the redesign pipeline | 2-3 concurrent requests |
| No cleanup of Playwright browser after error | Browser processes accumulate, ports exhausted | `async with async_playwright()` context manager, never bare `.launch()` without cleanup | After 5-10 failed redesigns |
| Unbounded `sites/` directory growth | Disk fills up over weeks/months | Automatic cleanup of results older than 7 days; track creation time | After ~500 redesigns |
| 2-second hardcoded sleep in crawler | Every redesign is 2s slower than necessary | Replace with `page.wait_for_load_state("networkidle")` or dynamic stability check | Noticeable from first request |
| 16K max_tokens truncating large redesigns | HTML output missing closing tags, broken pages | Increase to 32K+ or implement streaming accumulation with truncation detection | Complex sites with many sections |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| `sandbox="allow-scripts allow-same-origin"` on iframe | Equivalent to no sandbox; script can access parent DOM, cookies, storage | Use `allow-scripts` alone; serve preview from separate origin |
| No path traversal protection on `/sites/{subdomain}/{path}` | Attacker reads `_registry.json`, `.env`, or any file the server process can access | Validate subdomain with regex `^[a-z0-9-]+$`; resolve path and verify `is_relative_to(sites_dir)` |
| Serving LLM output HTML without sanitization | Prompt injection causes Claude to embed malicious script | Strip `<script>`, `<iframe>`, event handlers server-side with `nh3`; CSP on iframe blocks execution |
| Turnstile token validated client-side only | Bots bypass the widget and POST directly to the form endpoint | Server-side token verification is mandatory |
| No URL validation on redesign input | `file:///etc/passwd` or `javascript:` URIs sent to Playwright | Validate URL scheme (http/https only), validate domain resolves, reject private IPs (SSRF) |
| CORS `allow_origins=["*"]` on FastAPI | Any site can make authenticated requests to the API | Explicit origin allowlist; update when domain changes |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| No progress feedback during 30-60s wait | User thinks the site is broken, closes tab, never returns | Staged progress: "Crawling site..." -> "Analyzing design..." -> "Generating redesign..." with estimated time |
| Progress bar that is not actually connected to real stages | Bar fills linearly over 60s regardless of actual progress; erodes trust if it hits 100% and keeps waiting | Map SSE events to real pipeline stages; use indeterminate animation between stages |
| Showing an error after 45s of waiting with no way to retry | User wasted time and has no recourse | Auto-retry once silently; if second attempt fails, show error with one-click retry button that preserves the URL |
| Full-page iframe with no loading skeleton | Flash of blank white before redesign loads | Show a skeleton/shimmer that matches the viewport, then fade in the iframe content |
| Mobile viewport mismatch in iframe preview | Redesigned desktop HTML shown at mobile width looks broken | Use `<meta name="viewport">` in the redesigned HTML; add viewport toggle (desktop/mobile) in the preview UI |
| Contact form appears only at the bottom | Users who are impressed want to act immediately, but must scroll past the redesign | Sticky CTA bar or floating "Hire me" button visible alongside the preview |

## "Looks Done But Isn't" Checklist

- [ ] **SSE streaming:** Works on localhost but test through Cloudflare tunnel with real latency
- [ ] **iframe preview:** Open browser DevTools console; check that `parent.document` from iframe throws a cross-origin error
- [ ] **Contact form:** Submit 20 times rapidly from a script; verify rate limiting rejects most
- [ ] **Turnstile:** Disable JavaScript in browser; verify the form POST is rejected server-side without a valid token
- [ ] **URL validation:** Submit `file:///etc/passwd`, `http://169.254.169.254/latest/meta-data/`, and `javascript:alert(1)` as redesign URLs; all must be rejected
- [ ] **Mobile responsive:** Load the entire flow (landing -> progress -> result -> contact) on a real phone, not just browser devtools
- [ ] **Error recovery:** Kill the FastAPI server mid-redesign; verify the frontend shows a meaningful error within 10 seconds, not an infinite spinner
- [ ] **Path traversal:** Request `GET /api/sites/../_registry.json` through the preview server; verify it returns 404, not the file
- [ ] **Concurrent requests:** Open 3 browser tabs and submit redesigns simultaneously; verify queuing works and the Mac does not become unresponsive
- [ ] **Tunnel reconnection:** Restart cloudflared mid-session; verify the frontend reconnects or shows a clear "connection lost" message

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| SSE buffering through tunnel | MEDIUM | Switch from EventSource (GET) to fetch+ReadableStream (POST); update cloudflared config to named tunnel with http2 |
| iframe XSS escape | HIGH | Audit all stored redesigned HTML files for scripts; add server-side sanitization; rotate any exposed cookies/tokens; re-sandbox all iframes |
| Resource exhaustion from spam | LOW | Add `asyncio.Semaphore(1)` + IP rate limit; kill runaway Playwright processes; restart services |
| Contact form spam | LOW | Add Turnstile verification endpoint; honeypot field is a 10-minute frontend change |
| Proxy architecture wrong (Next.js routes) | HIGH | Rewrite all API calls from relative paths to direct FastAPI URLs; add CORS config; update tunnel ingress rules |
| Dropped SSE with no job ID | HIGH | Requires adding job tracking (ID, status, result storage) to FastAPI; rewriting frontend from fire-and-forget to poll-capable |
