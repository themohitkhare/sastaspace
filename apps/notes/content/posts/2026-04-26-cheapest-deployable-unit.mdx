---
title: A subdomain is the cheapest deployable unit I know.
date: 2026-04-26
summary: Containers are too heavy. Files are too light. A subdomain on a tunnel is the right size.
---

Anything that ships needs a deployable unit — the smallest thing you can stamp out and put on the internet. Big shops use clusters. Most people end up with VMs or containers. Static-site folks use folders.

For a lab, all of those are wrong sized.

A cluster is too much ceremony for a thing that takes a weekend to build. A container is the right shape but assumes you have somewhere to run it, plus a reverse proxy, plus DNS. A static-export folder is great until the project needs any state at all.

What I've ended up using: a subdomain on the existing Cloudflare tunnel. That's it. A new project means:

```
1. write a new entry in the tunnel ingress (one curl call)
2. add a CNAME for the subdomain (one curl call)
3. run the container behind it (docker compose up)
```

Three commands. The CI workflow that ships the project doesn't need to know about Cloudflare, doesn't need to know about TLS — that's all handled by the tunnel that's already there. The container doesn't need a reverse proxy because the tunnel routes by hostname directly to the container's port. There's no kubernetes manifest to argue with.

The unit cost of a new project: the time it takes to think of a name and write three curl commands. The time it takes to think of a name dominates.

I think this is the right size because every project here gets to be its own thing — its own URL that I can text to someone — without becoming its own ops surface. The shared infrastructure handles the boring parts; the subdomain handles the identity.

The bill at the end of the month doesn't change. Cloudflare doesn't charge per subdomain on a tunnel. The taxila box is the same regardless of how many containers it runs. New projects are essentially free, which is the whole point.
