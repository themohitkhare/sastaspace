# TypeWars — Player Auth (Guest → Verified) Design

**Date:** 2026-04-26
**Scope:** Add an optional sign-in tier to TypeWars. No mandatory accounts.

---

## Overview

TypeWars currently lets anyone play with just a callsign — Identity is generated client-side by the SpacetimeDB SDK, persisted in `localStorage`. Clearing site data wipes a player. There's no profile, no avatar, no cross-device continuity.

This spec adds an optional sign-in tier on top. Anyone can still join faceless and play; signing in promotes the player into a verified state with a persistent Identity (derived from email), a recognisable face (initials avatar with verified pip), and a profile. The sign-in flow reuses the existing magic-link service at `services/auth/` (the same one notes commenters use).

The spec is deliberately small. There is one new reducer, one new field on `Player`, one new route, and one shared avatar component. Nothing is mandatory; nothing breaks the existing guest experience.

---

## The Two States (one tier)

A single `Player` table; every row has the same primary key (Identity). What changes is *how* the Identity got there.

| | Guest | Verified |
|---|---|---|
| Identity source | SDK-autogenerated keypair, `localStorage` | Email-derived (deterministic via the auth service's existing JWT flow) |
| Persistence | Browser-local; cleared cookies = new player | Cross-device; same email = same Identity |
| `Player.email` | `None` | `Some("user@example.com")` |
| Avatar | Initials disc, legion color | Initials disc + verified pip (sasta-orange dot) |
| Leaderboard | Listed normally | Listed normally + visible verified pip |
| Profile modal | Stats only | Stats + verified-since timestamp |

A signed-in player on browser A who clears localStorage on browser B and signs in there ends up with the same Identity (the auth service derives Identity from email), so progress follows them across devices automatically. The `claim_progress` reducer is only invoked the *first* time a guest converts.

## Schema

One new field on the existing `Player` table:

```rust
pub struct Player {
    #[primary_key]
    pub identity: Identity,
    pub username: String,
    pub legion: u8,
    pub total_damage: u64,
    pub season_damage: u64,
    pub best_wpm: u32,
    pub joined_at: Timestamp,
    pub email: Option<String>,   // NEW: None = guest, Some(...) = verified
}
```

`email.is_some()` doubles as the verified flag. No `email_verified_at`, no `last_seen_at` — both YAGNI.

**Username uniqueness** is added to `register_player` going forward (case-insensitive). The two existing rows in prod (`smoketest`, `e2e_recruit`) have no conflict.

## Reducers

### New: `claim_progress(prev_identity, email)`

Public reducer, called by the frontend after a successful magic-link sign-in. Caller is the email-derived identity (`ctx.sender()`).

```rust
#[reducer]
pub fn claim_progress(
    ctx: &ReducerContext,
    prev_identity: Identity,
    email: String,
) -> Result<(), String> {
    let me = ctx.sender();
    let guest = ctx.db.player().identity().find(prev_identity);
    let existing = ctx.db.player().identity().find(me);

    match (guest, existing) {
        // First-time claim: rekey the guest row to the email identity.
        (Some(g), None) => {
            ctx.db.player().identity().delete(g.identity);
            ctx.db.player().insert(Player { identity: me, email: Some(email), ..g });
        }
        // Returning user with prior account on this email — merge guest into existing.
        (Some(g), Some(mut e)) => {
            e.total_damage = e.total_damage.saturating_add(g.total_damage);
            e.season_damage = e.season_damage.saturating_add(g.season_damage);
            e.best_wpm = e.best_wpm.max(g.best_wpm);
            e.email = Some(email);
            ctx.db.player().identity().delete(g.identity);
            ctx.db.player().identity().update(e);
        }
        // First-time sign-in, no prior guest play. Stamp email on existing row.
        (None, Some(mut e)) => {
            e.email = Some(email);
            ctx.db.player().identity().update(e);
        }
        // No guest, no existing — frontend should send them to legion-select after.
        (None, None) => {}
    }
    Ok(())
}
```

When the email row already has a different legion than the guest, the email row's legion wins (it's the "real" identity). Damages add. Best WPM is the max of either side.

### Updated: `register_player`

Add a username-uniqueness check (case-insensitive). Reject with an error string if any existing Player row's `username.to_lowercase()` matches the requested username's lowercase form. Existing guarantees (1–32 chars, legion 0–4) stay.

### No other reducer changes.

The auth service (`services/auth/`) does **not** need any new endpoints or privileged module calls. The existing `/auth/verify` flow already returns a SpacetimeDB JWT + Identity; the typewars frontend just needs to receive that callback.

## Frontend flow

1. **Sign-in trigger.** WarMap topbar gains a `sign in →` button (link-btn, next to `leaderboard →`) for guests. Verified players see their avatar disc with a tooltip — no menu.
2. **Modal.** Clicking "sign in" opens a modal with a single email input + "send magic link" button. The component is lifted from `apps/notes` (extracted into a shared package, e.g. `packages/auth-ui`, since notes uses the same modal). On submit it POSTs to `https://auth.sastaspace.com/auth/request` (existing endpoint) with `{ email, callback: "https://typewars.sastaspace.com/auth/verify" }`. Success state: "check your inbox".
3. **Magic link.** User clicks the email link → routes to `auth.sastaspace.com/auth/verify?t=<token>` → on success the auth service redirects to the callback URL with `?jwt=<jwt>&email=<email>` appended.
4. **Callback page.** New route `apps/typewars/src/app/auth/verify/page.tsx`. Rendered *outside* the `<SpacetimeDBProvider>` so it owns its own short-lived connection. On mount:
   - Read `jwt` and `email` from query params; bail with an error message if either missing.
   - Read `prevIdentity` = the existing guest identity from `localStorage.getItem('typewars:auth_token')` (or, if no token stored yet, the autogenerated identity the SDK would otherwise mint — derivable from `localStorage.getItem('spacetimedb:identity')` or by opening a one-shot anonymous connection and reading `ctx.identity`).
   - Build a one-shot `DbConnection` with `.withToken(jwt)`. On connect: call `reducers.claim_progress({ prevIdentity, email })`, await its reducer-event acknowledgement, then disconnect.
   - `localStorage.setItem('typewars:auth_token', jwt)`. Navigate to `/`.
5. **Done.** The home page re-mounts the provider, which reads the new token and connects as the email-derived identity. The player subscription returns the now-merged row; warmap renders normally.

## UI components

### `<Avatar callsign legion verified size? />`

Reusable component, single file `apps/typewars/src/components/Avatar.tsx`. Renders a circular disc:
- Background: legion color (`LEGION_INFO[legion].color`).
- Foreground: first 1–2 letters of `callsign` (skip non-letters, fallback to first character), white, mono font, weight 500.
- Border: 1px `var(--brand-dust-40)`.
- Size: default `24px`, prop override (e.g. `48px` for the profile modal).
- If `verified`: 6px disc in `var(--brand-sasta)` with a 1px `var(--brand-paper)` border, absolutely positioned bottom-right.

Used in: WarMap topbar player pill, Battle header (next to "back" button), Liberation splash contributor rows, Leaderboard player roster, Profile modal header.

### Sign-in modal (`<SignInModal />`)

Lifted from `apps/notes/src/components/AuthMenu` patterns. Lives in `packages/auth-ui` so notes and typewars share. Email input, submit button, success / error states. Closes on backdrop click.

### Profile modal (`<ProfileModal player />`)

Opens when a callsign is clicked anywhere it appears (Leaderboard player roster row, Liberation contributors list). Renders:
- 48px avatar (verified pip if applicable).
- Callsign + verified badge.
- Legion banner (mech name + colour bar).
- Joined-at (e.g. "joined 2 days ago").
- 4 stat chips: total damage, season damage, best WPM, regions held.
- No "follow" or social actions for MVP.

## CSP / Infrastructure

Nothing new. The current shared `infra/landing/security_headers.conf` already allows `connect-src https://auth.sastaspace.com` (used by notes commenters). The magic-link request and verify both go through that domain.

The typewars backend module needs to be republished after adding the new reducer + schema field; bindings need regeneration via `pnpm typewars:bindings`. Standard backend-deploy CI path covers it.

## Out of scope (explicit)

These are deliberately not in this spec; can be added later if there's demand:

- **Sign-out button.** To return to guest, clear site data. No UI affordance.
- **Email change.** Sign in with a new email is treated as a separate account; the old account is orphaned but its damage stays on the legion total.
- **Two-account merging** (other than guest → email-on-claim). Not supported.
- **Username editing.** Immutable after `register_player`. Aligns with "Legion is permanent" theme.
- **Stale JWT auto-refresh.** When a JWT expires the SDK falls back to a fresh anonymous identity. User signs in again.
- **Anti-spam on magic-link requests.** Already handled by the existing auth service rate limit; nothing typewars-specific.
- **Multiple browsers concurrently signed in.** Works automatically — both share the same email-derived Identity. No multi-session UX needed.
- **Cross-device sign-in carrying guest progress.** If a user plays as guest on browser A but clicks the magic link in browser B, browser B has no `prevIdentity` to claim — guest progress on A is orphaned. The sign-in modal should hint "click the email link in this same browser to keep progress" but enforcing it is out of scope.
- **Public profile route** (e.g. `/p/<username>`). Profile modal is enough for MVP. Add a route the day someone wants to share their profile.
