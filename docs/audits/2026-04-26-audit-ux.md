# UX / Interaction Audit — 2026-04-26

**Auditor:** ux
**Methodology:** Static code + E2E spec analysis across all four apps (`landing`, `notes`, `typewars`, `admin`). Read every component in the auth, comment, admin queue, deck, and typewars battle flows end to end. Examined CSS modules for focus/touch/responsive coverage. Cross-referenced committed e2e specs (`auth.spec.ts`, `notes-auth-stdb.spec.ts`, `comments-*.spec.ts`, `admin.spec.ts`, `admin-panels.spec.ts`, `deck.spec.ts`, `moderator.spec.ts`) to identify gaps between spec intent and implementation.

**Overall grade:** 6 / 10

The craft on the "golden path" flows is genuinely good — error copy, magic-link UX, and the deck stepper are all well-considered. The grade is pulled down by three systemic gaps: no focus management in any of the three modal variants, zero mobile support for the typewars battle grid, and the total absence of keyboard traps, aria roles, and focus rings outside the notes auth modal.

---

## Per-flow walkthrough

- **Sign in to notes (magic link STDB-mode):** Solid happy path. `AuthMenu` focuses the email input on open, handles Esc, shows inline errors, and the "sent" state explains the 15-minute window. Gap: the sent state offers no resend option, and wrong-email correction requires closing and re-opening the modal.
- **Post a comment as a signed-in user:** Correct — sign-in gate is shown to anon users; signed-in users see `posting as <name>`, the form validates on length (4–4000 chars), submission shows "sending…", and success shows a moderation-pending confirmation with a "write another →" reset. No optimistic rendering — the comment does not appear in the list until the moderator approves, which is by design but not explained in the post-submit message.
- **Sign in to typewars + claim a player:** Functional. The `SignInModal` from `packages/auth-ui` auto-focuses the email input. The identity-claim (`prevIdentity`) hint is shown only when `prevIdentity` is set; the callsign/legion form has inline error display and a disabled "ENLIST →" state. Gap: no Esc binding on the `SignInModal`, no aria role or `aria-modal` attribute, no focus trap.
- **Play a battle round:** The battle screen `autoFocus`s the text input on mount — correct. Live word highlighting, streak/multiplier HUD, HP bar, and the shake/flash feedback are all implemented. Gap: the screen is an SVG + grid layout that does not adapt to mobile viewports (the word grid is hard-coded at 4×2, no touch keyboard accommodation, no way to see words when the soft keyboard steals 40% of screen real estate).
- **View leaderboard:** Renders correctly. Clicking a row opens `ProfileModal`. Gap: the leaderboard rows use `role="button"` + `onClick` but have no `tabIndex` or `onKeyDown` — keyboard users cannot navigate rows; the modal itself has no `role="dialog"` or `aria-modal`.
- **Generate a deck plan:** The three-step flow (describe → plan → generate+download) is the strongest flow in the codebase. Progress animation, example prompts, per-track editing with segmented selectors, and the live musicgen prompt preview are all thoughtful. The download button mutates its own label ("building zip… → downloaded ✓") as feedback. Gap: the "share link" button copies nothing (the handler only mutates the label; there is no `navigator.clipboard.writeText` call) — it is dead UI.
- **Owner approves a pending comment in admin:** Strong for the admin app. The `Comments` panel uses optimistic updates (`setOptimistic`) so the status badge flips immediately; the spinner `actioning` state prevents double-clicks. A confirmation modal guards destructive delete. Empty-state copy is specific per filter. Gap: admin auth errors silently swallow exceptions (`catch { /* ... */ }`), so a failed approve gives no feedback to the owner beyond the badge reverting.
- **Moderator-flagged comment surfaces in admin:** The `verdictMap` correctly surfaces the moderator agent's reason string on flagged rows (`verdict: injection`). The admin `Comments` panel shows it in a secondary metadata line. This flow is well-handled.

## Critical findings

### C1 — Battle screen is unusable on mobile (typewars)
The word grid is `grid-template-columns: repeat(4,1fr); grid-template-rows: repeat(2,1fr)` with a `min-height: 360px` and no breakpoint. The `@media (max-width: 720px)` block does not touch the battle screen at all. On a phone the 8-word grid is crushed to tiny unreadable cards, the `battle-input` fires the soft keyboard which then obscures most of the game area, and the 4×2 layout is impossible to play. The entire battle flow is a no-op on mobile.

**File:** `apps/typewars/src/styles/typewars.css` lines 260-264, `@media` block at line 408 (omits all battle classes).

### C2 — Modal focus trap missing in every modal variant
Three modal implementations exist: `AuthMenu.modal` (notes), `SignInModal` (auth-ui package), and the typewars `modal-backdrop`. None implement a focus trap. Tab keystrokes escape the modal into the page background, which is both an a11y failure (WCAG 2.1 SC 2.1.2) and a UX bug (users can accidentally interact with obscured content).

**Files:** `apps/notes/src/components/AuthMenu.tsx` (has Esc ✓, no trap), `packages/auth-ui/src/SignInModal.tsx` (no Esc, no trap), `apps/typewars/src/components/LegionSwapModal.tsx`, `ProfileModal.tsx` (no Esc, no trap).

### C3 — "Copy share link" button on deck is dead UI
In `Deck.tsx` `Results` component, the share button handler sets `shareLabel("link copied")` after a timeout but never reads the current URL or calls `navigator.clipboard.writeText`. Clicking it does nothing except show misleading copy.

**File:** `apps/landing/src/app/lab/deck/Deck.tsx` lines 966-972.

## High-priority findings

### H1 — No resend / wrong-email path in the magic-link sent state
After `AuthMenu` reaches `modal.kind === "sent"`, the only affordance is a "close" button. There is no "resend" option, no "wrong email? → change it" link, and no explanation that the sent state is permanent (closing and re-opening the modal resets `email` state because `email` is local to the component, so the user must retype). The 15-minute window is mentioned but the user cannot act within the same modal.

**File:** `apps/notes/src/components/AuthMenu.tsx` lines 107-120.

### H2 — `SignInModal` (auth-ui) missing Esc close, aria-modal, and role="dialog"
The shared `SignInModal` used in typewars has no Esc handler, no `role="dialog"`, no `aria-modal="true"`, and no `aria-label`. The notes `AuthMenu` has all of these; the shared package is a regression.

**File:** `packages/auth-ui/src/SignInModal.tsx` lines 70-106.

### H3 — Admin error handling silently drops failures
In the admin `Comments` panel (`setStatus`) the catch block calls `setOptimistic(prev => { const n = new Map(prev); n.delete(id); return n; })` but shows no toast, banner, or inline message to the owner. The badge reverts without explanation. Similarly `doDelete` catches with `/* row stays */` — the owner sees no feedback when a delete fails.

**File:** `apps/admin/src/components/panels/Comments.tsx` lines 108-111.

### H4 — Legion swap modal does not call any reducer — swap is silently ignored
`LegionSwapModal`'s `onSwap` prop receives the chosen `LegionId` and calls `setSwapOpen(false)` in App.tsx. The reducer call to actually swap the player's legion is missing: `swapLegion` in `App.tsx` just calls `setSwapOpen(false)` and discards the `_legion` argument. A player who clicks "CONFIRM SWITCH →" gets no error, the modal closes, and nothing changes server-side.

**File:** `apps/typewars/src/components/App.tsx` lines 68-70 (`swapLegion` callback body).

### H5 — Deck "generating" phase is fully fake (no real generation round-trip in legacy mode)
In the legacy HTTP path without `NEXT_PUBLIC_DECK_API_URL`, `stubZipBlob()` returns a text file, and `GeneratingView` is a client-side animation that calls `onDone` when its internal timer completes. The user sees all tracks "processing" and "done" but no audio is produced. The "downloaded ✓" state shows misleading success. This is documented as "offline-prototype mode" in a code comment but is the default state for new deployments and there is no visible hint to the user that the download is a placeholder.

**File:** `apps/landing/src/app/lab/deck/Deck.tsx` lines 1470-1483.

## Medium

### M1 — Comment list shows no loading skeleton — snaps from empty to loaded
`Comments.tsx` initialises `comments` as `null` and renders the empty-state copy ("No comments yet. Be the first to break the silence.") during the STDB WebSocket connection phase, then snaps to the real list on `onApplied`. A brief skeleton or spinner between `null` and `[]` would prevent the premature empty-state flash.

**File:** `apps/notes/src/components/Comments.tsx` lines 30-31.

### M2 — Admin connect error is only `console.warn`
`subscribeAdminComments` in `admin.ts` calls `.onConnectError` and logs a warning but never surfaces it to the UI. If the STDB connection fails, the admin page shows "connecting…" indefinitely.

**File:** `apps/notes/src/lib/admin.ts` line 74.

### M3 — Leaderboard rows keyboard-inaccessible
`lb-trow` rows in the leaderboard use `onClick` + `role="button"` but have no `tabIndex` and no `onKeyDown` handler. Keyboard users cannot open player profiles.

**File:** `apps/typewars/src/components/Leaderboard.tsx` lines 139-143.

### M4 — Deck textarea has no accessible label
In `Composer`, the textarea has no `id`, no `<label>`, and no `aria-label`. The visible label above it (`"describe what you're building"`) is a presentational `div` with class `composerLabel`. Screen readers have no way to associate the label with the input.

**File:** `apps/landing/src/app/lab/deck/Deck.tsx` lines 418-434.

### M5 — Callsign input in LegionSelect has no maxLength validation feedback
The input has `maxLength={24}` which silently truncates. There is no character counter or error message when the user hits the limit, unlike the deck textarea which shows `{value.length} / 600`.

**File:** `apps/typewars/src/components/LegionSelect.tsx` lines 63-70.

### M6 — Admin routing uses hash fragments — no shareable deep links
The admin shell routes via `window.location.hash` (e.g. `#/comments?status=pending`). Hash-based routing does work for deep-linking but prevents prefetch and makes the URL less legible. More practically, the `Comments` panel's `initialFilter` param is read from `route.params` at mount only; navigating directly to `admin.sastaspace.com/#/comments?status=flagged` requires the shell to have parsed the initial hash, which it does via `parsePath()` in the lazy initializer — this is actually correct. Low severity.

### M7 — Deck track editor inputs have no `id`/`htmlFor` pairing
In `PlanEditor`, the `<label>` elements use a `className="fieldLab"` and sit visually adjacent to their inputs but are not linked via `htmlFor`/`id`. Clicking the label text does not focus the input.

**File:** `apps/landing/src/app/lab/deck/Deck.tsx` lines 698-756.

### M8 — `AuthMenu` error state does not announce itself to screen readers
The error paragraph `{modal.kind === "error" && <p className={styles.error}>{modal.message}</p>}` is rendered but has no `role="alert"` or `aria-live="polite"` attribute. Screen readers will not announce the error automatically after form submission.

**File:** `apps/notes/src/components/AuthMenu.tsx` lines 143-145.

### M9 — Admin sign-in loading state is a bare `<div class="spinner"/>` with no text
During initial auth state resolution (`authState === "loading"`), the admin renders only a spinner div with no accessible text and no label. Screen readers would announce nothing.

**File:** `apps/admin/src/components/Shell.tsx` line 147.

## Mobile UX-specific issues

1. **Battle screen unusable on mobile** (see C1 — critical). The entire typewars battle flow needs a mobile layout: 2×4 or scrollable word list, sticky input bar above the soft keyboard, HP/stats at top only.

2. **Auth modal aligns correctly on mobile** — the notes `auth.module.css` includes `@media (max-width: 640px)` that moves the modal to `align-items: flex-start; padding-top: 80px` and bumps the email input to `font-size: 16px` (prevents iOS zoom). This is done correctly.

3. **Comment form stacks correctly on mobile** — `comments.module.css` has a `640px` breakpoint that stretches the submit button full width and stacks actions. Correct.

4. **Deck page has no mobile breakpoint** — `deck.module.css` does not contain any `@media` rules at all. The 52px hero heading, fixed-width stepper, and wide plan layout will overflow on phones. The 1040px max-width container uses `padding: 0 32px` which is acceptable on tablets but gives only 16px headroom on small phones if not reduced.

5. **Typewars legion grid collapses correctly** — the `@media (max-width: 720px)` block includes `legion-grid { grid-template-columns: 1fr }` and enlist button sizing. Mobile enlistment works.

6. **Map war map: no touch gesture for globe rotation** — the map drag uses mouse events (`mousemove`, `mouseup`). Touch events (`touchstart`, `touchmove`, `touchend`) are not handled. On mobile, dragging the globe does nothing, and region selection relies on tap (which does work via `onClick`).

## a11y findings

1. **No `lang` attr gap:** All four `layout.tsx` files include `<html lang="en">` — correct.

2. **Notes nav landmark:** `<nav aria-label="Primary">` present in `TopBar.tsx` — correct. Landing `TopNav` also has `aria-label="Primary"` and `aria-current="page"` on active items — correct.

3. **Modal role/aria-modal gap:** Notes `AuthMenu` has `role="dialog" aria-modal="true" aria-label="Sign in"` — good. The typewars modal-backdrop (`.modal-backdrop`) and `SignInModal` have none of these. `LegionSwapModal` and `ProfileModal` also lack `role="dialog"` and `aria-modal`.

4. **Focus ring absent in typewars:** The only `:focus` rule in `typewars.css` is on `.callsign-input`. All interactive elements (legion cards, region list rows, back button, leaderboard rows) use custom hover styles but no `:focus-visible` outline. Keyboard users get no visible focus indicator.

5. **Focus ring present in notes:** `auth.module.css` uses `outline: none` with a `border-color` replacement on focus — visible but non-standard. `comments.module.css` does the same. These are keyboard-visible but use color alone, which fails SC 1.4.1 (use of color) for users who can't distinguish the border color change.

6. **Images / SVGs:** All inline SVGs in the battle screen, track player, and nav use `aria-hidden="true"` on decorative SVGs and `aria-label` on functional buttons (play, pause, regenerate, download). This is correctly implemented.

7. **Canvas waveform in deck results:** The `<canvas>` element has no accessible label or description. For screen reader users, the waveform is invisible and the track cannot be seeked accessibly. At minimum `role="img" aria-label="Waveform for {t.name}"` should be added.

8. **`comments` section landmark:** `<section aria-label="Comments">` in `Comments.tsx` — correct.

9. **Admin panel navigation:** The sidebar nav uses `<button>` elements for all nav items (correct; not `<a>` for a SPA), but collapsed sidebar state only shows icons with `title` tooltip — not surfaced to screen readers (`title` is not a reliable accessible name). Icon buttons need `aria-label`.

10. **Route-change focus management:** Notes is server-side-rendered (Next.js page router) — focus management on navigation is handled by the browser + Next.js router. TypeWars is a single-screen SPA with client-side "screen" state changes; screen transitions (`setScreen('battle')`) do not move focus to the new screen's heading or input. After the battle loads, `autoFocus` on the input is the only focus placement, which is technically adequate but not ideal.

## Recommended next 5 actions

1. **Fix the LegionSwapModal: wire the reducer.** The swap flow completing silently without actually swapping is the single most impactful broken interaction. In `App.tsx`, `swapLegion` must call `changePlayerLegion` (or equivalent reducer) with the chosen id before closing the modal. Add a loading state and an error banner on failure.

2. **Add focus trap to all three modal implementations.** Implement a `useFocusTrap` hook (or use a library like `focus-trap-react`) and apply it to `AuthMenu.modal`, `SignInModal`, `LegionSwapModal`, and `ProfileModal`. Also add `role="dialog"` + `aria-modal="true"` + `aria-label` to the typewars modals, and an Esc handler to `SignInModal`.

3. **Fix the "copy share link" button on deck.** Replace the current label-only handler with `navigator.clipboard.writeText(window.location.href)` (or a constructed share URL). If there is no shareable URL concept yet, remove the button rather than leaving dead UI.

4. **Design a mobile layout for the typewars battle screen.** The 4×2 word grid needs a single-column scrollable fallback at ≤720px, the input row needs to be sticky above the soft keyboard (use `position: fixed; bottom: 0` or `padding-bottom: env(keyboard-inset-height)` via CSS viewport API), and touch events need to be added to the globe's drag handler for the war map.

5. **Add `:focus-visible` outlines and `role="alert"` to error states.** Apply a 2px offset focus-visible ring on all interactive elements in typewars (legion cards, region rows, leaderboard rows). Add `role="alert"` to the `AuthMenu` error paragraph and the admin `Comments` panel failure feedback so screen readers announce errors on submission.
