### HIGH

#### H1 — `User.email` leaked to every connected client via public table

**Location:** `modules/sastaspace/src/lib.rs:61–68`

**Issue:** The `User` table is declared `#[table(accessor = user, public)]`. In SpacetimeDB, `public` means every connected client's subscription can read all rows including all columns. The `email` field is therefore readable by any client that subscribes to the `User` table. The doc-comment on line 58 ("It is NEVER surfaced to other clients via the public schema") is incorrect — the schema does surface it.

**Exploit scenario:** Any visitor who opens a WebSocket connection to `wss://stdb.sastaspace.com` and subscribes to `SELECT * FROM user` receives every registered user's email address, display name, and STDB identity in plaintext. With ~15 lines of SpacetimeDB SDK code, an anonymous actor can harvest the complete user list.

**Proposed fix (two options, pick one):**  
Option A — Remove the `email` field from the public `User` table. Store it in a separate private table keyed on identity (e.g. `UserEmail { identity, email }` with no `public` attribute). The `register_user` and `verify_token` reducers write to both; the `email` table is owner-read-only.  
Option B — Keep the unified table but add a column-level exclusion. SpacetimeDB 2.1 does not natively support per-column visibility; so this requires splitting the table (same as Option A in practice).

**Effort:** ~2 hours. Schema migration + update `register_user`, `verify_token`, `consume_auth_token`, and any SQL that reads `user.email`.

#### H2 — `log_event` (public table) streams live container logs to any subscriber

**Location:** `modules/sastaspace/src/lib.rs:1234–1245`

**Issue:** `log_event` is `#[table(accessor = log_event, public)]`. The admin-collector appends lines from `docker logs --follow` for every container the owner has shown interest in. Any connected client subscribing to `SELECT * FROM log_event` receives these lines in real time.

**Exploit scenario:** A registered user (or any anonymous WebSocket client) subscribes to `log_event` and reads application logs that may contain stack traces, internal endpoint paths, email addresses being processed, Resend API error bodies (which include recipient emails), or SpacetimeDB identity hex values. This is an information-disclosure path that could assist targeted attacks against the running infrastructure.

**Proposed fix:** Remove the `public` attribute from `log_event`. Access is already owner-gated at the write side (`assert_owner` in `append_log_event`, and `add_log_interest` is owner-only). Making the table non-public restricts subscriptions to the owner identity. The admin TUI panel still works because it connects as the owner.

**Effort:** 1-line change in `lib.rs` + republish module. No schema migration required (the table shape is unchanged; only the subscription filter changes).

### MEDIUM

#### M1 — `request_magic_link` has no per-email or per-identity rate limit; any anonymous caller can trigger unbounded Resend sends to arbitrary addresses

**Location:** `modules/sastaspace/src/lib.rs:670–733`

**Issue:** `request_magic_link` is callable by any unauthenticated identity (no `assert_owner`, no user-membership check). It inserts one `auth_token` row and one `pending_email` row per call, with no check for how many tokens are already pending for the target email. A bot making 1000 calls with `email = "victim@example.com"` would queue 1000 Resend sends before the 15-minute TTL expires, exhausting your Resend free-tier quota and sending 1000 emails to the victim.

**Exploit scenario:** Resend's free tier has a 3000 email/month cap. A one-minute attack loop saturates that budget and disables all future magic-link auth for legitimate users. Email receivers also see this as spam and may mark the domain; Resend may suspend the sending domain.

**Proposed fix:** Before inserting the new `auth_token` row, count how many unexpired, unused tokens already exist for `email`. Reject if the count ≥ some small cap (e.g. 3). Example:

```rust
// Inside request_magic_link, before the insert:
let active_count = ctx
    .db
    .auth_token()
    .iter()
    .filter(|t| t.email == email && t.used_at.is_none()
        && !is_token_expired(now.to_micros_since_unix_epoch(),
                             t.expires_at.to_micros_since_unix_epoch()))
    .count();
if active_count >= 3 {
    return Err("too many pending tokens for this email; wait 15 minutes".into());
}
```

`auth_token` is indexed by token (primary key) but not by email. Add `#[index(btree)]` to `auth_token.email` so this scan is O(log n) rather than a full table walk.

**Effort:** ~1 hour.

#### M2 — `verify_token` is callable by any unauthenticated identity; token brute-force is theoretically possible but bounded by entropy

**Location:** `modules/sastaspace/src/lib.rs:775–818`

**Issue:** `verify_token` is open to any caller; there is no lockout after N failed attempts. Tokens are 32 alphanumeric characters (62^32 ≈ 2^190 space), so brute-force is computationally infeasible. However, there is no counter-pressure if SpacetimeDB itself enforces no per-connection call rate limit. In a future where token entropy is inadvertently reduced (e.g. RNG seeded with a low-entropy value in a test mode), this becomes exploitable.

**Exploit scenario:** Not currently exploitable given the token space. The risk is residual: a failed call leaks one bit of information ("unknown token" vs "expired" vs "already used"). An attacker who somehow learned a token was issued for a specific email (e.g. from the public `pending_email` table if it were ever made public — it is not currently) could attempt targeted guesses.

**Proposed fix:** SpacetimeDB 2.1 does not expose per-connection call metrics natively. Add a `failed_auth_attempt` counter table (private, keyed on caller identity) that increments on `verify_token` failure and rejects the call after N failures within a rolling window. This is low priority given the entropy — document it as a v2 hardening item. More immediately, add an `#[index(btree)]` on `auth_token.email` (same as M1 fix) to make any future expiry scan efficient.

**Effort:** 2–3 hours if implemented; accept-and-track is fine at this threat level.

#### M3 — JWT replay window for owner actions: stolen `id_token` usable for up to ~1 hour

**Location:** `modules/sastaspace/src/lib.rs:191–220` (`verify_owner_jwt` / `set_comment_status` / `delete_comment`)

**Issue:** `verify_owner_jwt` decodes the JWT payload without checking the `exp` claim. A Google device-flow `id_token` has a ~1 hour expiry. If the token is captured from the TUI keychain (requires local access to the machine) or from TLS traffic (requires MITM), the attacker can call owner-gated reducers (`set_comment_status`, `delete_comment`) for up to one hour, combined with the owner STDB identity token also being compromised. This is a defense-in-depth gap, not a standalone exploit — `assert_owner` already requires the STDB identity key.

**Exploit scenario:** An attacker who has both the STDB owner token AND a captured Google `id_token` (both from the same machine) can act as the owner for up to one hour after the `id_token` was issued, even if the user has revoked the session in Google's UI. Combined risk is low given the two-factor requirement.

**Proposed fix:** Add `exp` claim validation to `jwt_email_claim`. The claim is already in the JWT payload (base64url-decoded inline); add a check after email extraction:

```rust
let exp = obj.get("exp").and_then(|v| v.as_i64()).unwrap_or(0);
let now_secs = ctx.timestamp.to_micros_since_unix_epoch() / 1_000_000;
if now_secs > exp {
    return Err("jwt expired".into());
}
```

**Effort:** ~30 minutes.

### LOW

#### L1 — GitHub Actions use tag-pinned actions (`@v6`, `@v7`) instead of SHA-pinned

**Location:** `.github/workflows/release.yml` (all `uses:` steps)

**Issue:** All action references use floating tags (e.g. `actions/checkout@v6`). A tag can be moved to point at a different commit by the action maintainer (or a compromised maintainer). SHA pinning (e.g. `actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683`) is the supply-chain-safe alternative.

**Impact:** Low. GitHub's own `actions/*` are maintained by GitHub and are low-risk in practice. The real risk is third-party actions, of which there are none here.

**Proposed fix:** Run `pinact` or manually SHA-pin each action. Not urgent.

---

#### L2 — Release binary signed by SHA-256 checksum only; no GPG or Sigstore/cosign signature

**Location:** `.github/workflows/release.yml` (cargo-dist autogenerated)

**Issue:** cargo-dist produces `sha256-sums` checksums for release archives but no GPG or Sigstore cosign signature. The Homebrew formula at `themohitkhare/homebrew-sastaspace` references the release by sha256 hash, which provides integrity but not authenticity (an attacker who controlled the GitHub release could replace the binary AND update the hash).

**Impact:** Low. GitHub release authenticity depends on GitHub account security. For a btop-style personal tool with a small known user base this is acceptable.

**Proposed fix:** Add cosign signing via `actions/cosign-installer` in the release workflow. cargo-dist has native cosign support. Track as a v2 release hardening item.

---

#### L3 — `submit_user_comment` stores `author_name` from `User.display_name` without length re-validation at insert time

**Location:** `modules/sastaspace/src/lib.rs:400–424`

**Issue:** `display_name` is validated to ≤64 chars on registration (`validate_register_user_inputs`). `submit_user_comment` reads it from the `User` row and stores it directly in `Comment.author_name` with no re-check. If the validation constraint were ever relaxed or bypassed (e.g. via a future `update_display_name` reducer), a long name could flow into `Comment`. Not a current exploit path.

**Proposed fix:** Assert `author_name.len() <= 64` at comment insert, or (simpler) leave it and add a `update_display_name` reducer that re-validates. Note as future-proofing.

## TUI Auth Review

**Magic-link flow (`crates/auth/src/magic_link.rs`):** Token is emailed as a raw 32-char alphanumeric string. The user pastes it into the TUI. The token travels over TLS (STDB HTTP API). After `verify_token` succeeds, the STDB bearer token is stored via the `keyring` crate (OS keychain on macOS = Keychain, Linux = libsecret). This is the correct approach; the token is not written to disk in plaintext. No issues found beyond the general replay window noted in M3.

**Google OAuth device flow (`crates/auth/src/google_device.rs`):** The client receives a Google `id_token` (JWT) and passes it to the STDB reducer. The client does NOT verify the JWT signature — it passes the token verbatim. The module's `verify_owner_jwt` does email-claim validation + `assert_owner` (see M3). The client-side non-verification is acceptable: the token was obtained by authenticating directly with Google over TLS, so the chain of custody is sound. The missing piece is expiry enforcement (M3).

**TUI tracing logs:** `tracing` output goes to `~/.local/share/sastaspace/logs/`. Reviewed `magic_link.rs` and `google_device.rs`: the only `info!` calls log the email and `user_code` (which is a short human-readable code, not a secret). The STDB bearer token and the Google `id_token` are NOT logged. No PII leak in the current logging surface.

## Workers Review

**`auth-mailer.ts`:** `RESEND_API_KEY` is correctly loaded from environment (Zod-validated, worker exits if absent). No hardcoded keys. The key is injected at deploy time via `deploy.yml` from a GitHub Secret. No issues.

**`moderator-agent.ts`:** LLM prompt injection is addressed via a two-layer architecture (injection detector first; classifier only runs on `BENIGN` input; content is wrapped in unique sentinel delimiters). Both parsers are fail-closed (`ATTACK` / `flagged` on any unexpected reply). Design is sound. The one gap: the sentinel strings (`<<<sastaspace_comment_8f3a2>>>`) are public in this open-source repo, so a sophisticated attacker could craft a comment that embeds the exact sentinel to attempt to escape the delimiter. The guards doc (`infra/agents/GUARDRAILS.md`) should note this and the adversarial test suite should include this case if it doesn't already.

**`admin-collector.ts`:** The Docker socket (`/var/run/docker.sock`) is mounted into the container at runtime (default `new Docker()` path). This is a known high-privilege bind-mount — any process inside the container that reads it can control the Docker daemon. The collector uses the socket read-only (listing containers, reading stats/logs), but the socket itself grants full daemon access. This is an accepted architectural constraint for an admin collector on a single-operator machine (taxila). Not classified as a finding since the threat model is: attacker needs container escape + socket access, at which point the game is already over. **Document it in the container's README as an explicit trust boundary.**

## Infrastructure Notes

- `stdb.sastaspace.com` — SpacetimeDB raw HTTP is exposed. All write-path reducers are either owner-gated or validated. The public-table leak (H1, H2) is the primary risk.
- `taxila` prod box — no k8s, docker-only. Attack surface is Cloudflare tunnel → nginx → STDB. Workers run with docker.sock access (see above). Container images are pulled on deploy, not pinned by digest — a future hardening item.
- `nginx:alpine` on port 3110 — static files only; no dynamic execution. Path traversal risk is nginx-standard. No custom config reviewed (not in this repo).
