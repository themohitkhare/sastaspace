# UI / Visual Design Audit — 2026-04-26

**Auditor:** ui  
**Methodology:** Static code review across all four user-facing apps (landing, notes, typewars, admin) plus the shared `packages/design-tokens` and `packages/auth-ui` packages. Inspected every CSS file, every layout.tsx, a representative sample of component TSX files (Avatar, Battle, Leaderboard, LegionSelect, LegionSwapModal, ProfileModal, AuthMenu, Comments, Shell, AuthSignIn, Chip, Icon, SignInModal), and cross-referenced against the brand guide embedded in `colors-and-type.css`. No browser was opened; all findings are from source analysis.

**Overall grade:** 7 / 10

The token foundation is genuinely strong — five colours, two families, a clean scale, one ease curve, no gradients or shadows (almost). Where the system breaks down is (a) the admin app duplicating the token file instead of importing it, (b) a 20-token-plus herd of unregistered status colours living only in admin's global CSS, (c) 103 inline `style={}` blocks in typewars that should be CSS classes, (d) the `box-shadow` brand prohibition being violated in two game-critical states, and (e) `font-weight: 600` appearing in exactly the three topbar instances that get the most eyeball time. The bar for "premium" is within reach; these are fixable issues, not structural ones.

## Design system coherence

### Token source of truth

`packages/design-tokens/src/colors-and-type.css` is well-authored: five primitive colours, semantic aliases, two font families, a 9-step scale, one ease curve, two durations, four radii, five space steps. The comment block at the top even documents the WCAG AA math (Rust on Paper = 8.1:1).

**Import discipline across apps:**

| App | How it gets tokens |
|---|---|
| landing | `@import "@sastaspace/design-tokens/colors-and-type.css"` — correct |
| notes | Same — correct |
| typewars | Same — correct |
| admin | **Copy-paste.** The entire `:root` block is hand-replicated in `globals.css` lines 7–69. If a token value changes in the package, admin diverges silently. |

### Naming inconsistency: two font-variable conventions

Landing, notes, and typewars use `--font-inter` / `--font-jetbrains` as the Next font CSS variables, then remap in `:root`:
```css
--font-sans: var(--font-inter, …);
--font-mono: var(--font-jetbrains, …);
```

Admin's `layout.tsx` binds fonts **directly** to `--font-sans` / `--font-mono`:
```ts
const inter = Inter({ variable: '--font-sans' });
```
This works but means admin cannot use the `--font-inter` / `--font-jetbrains` intermediary variables if it ever imports the shared token file — it would need an extra `:root` remap block. The convention should be standardized.

### Tailwind @theme duplication

Landing and notes both define a `@theme` block re-asserting the brand colours as Tailwind v4 primitives (e.g., `--color-sasta: #c05621`). These raw hex values are hardcoded in each `@theme` block separately — they do not reference the already-imported CSS variable. This is a minor but real drift hazard.

### Unregistered status colours

Admin's `globals.css` contains 15+ hardcoded hex values for status states (`#b8412c`, `#4a7c3f`, `#a86a17`, `#fbe9e3`, `#fdf3df`, `#e6efe1`, `#f1e2bf`, `#d2e1ca`, `#d4a13b`, `#d04a3b`, `#f0cfc4`) that are not represented in the design-token file. Some of these are also repeated in TSX files (`Server.tsx`, `Logs.tsx`, `TypeWars.tsx`, `AuthDenied.tsx`). The green, yellow, red trio is used in at least four distinct places with slightly different raw values (`#a86a17` vs `#a86a17` — actually consistent — but `#d04a3b` for the nav dot vs `#b8412c` for danger text are definitely different reds for the same semantic meaning). These should be `--color-status-danger`, `--color-status-warn`, `--color-status-ok` tokens.

## Critical findings

### C1 — Admin duplicates the design-token file instead of importing it

**File:** `apps/admin/src/app/globals.css` lines 7–69  
**Risk:** Every token change in `packages/design-tokens` requires a manual mirror edit in admin. One missed update and the admin panel renders with stale colours. This is a divergence time-bomb, not a theoretical concern.  
**Fix:** Add `@sastaspace/design-tokens` to admin's `package.json` devDependencies, replace the hand-rolled `:root` block with `@import "@sastaspace/design-tokens/colors-and-type.css"`.

### C2 — 15+ status colours are off-system tokens used in both CSS and TSX

**Files:** `apps/admin/src/app/globals.css`, `apps/admin/src/components/panels/Server.tsx`, `apps/admin/src/components/panels/Logs.tsx`, `apps/admin/src/components/panels/TypeWars.tsx`, `apps/admin/src/components/auth/AuthDenied.tsx`  
**Risk:** The danger red appears as `#d04a3b` (nav pulse dot), `#b8412c` (chip/button danger text), and `#b8412c` again in TSX inline styles — two different reds for the same semantic meaning. If the design shifts these colours, 8+ call sites need manual hunting.  
**Fix:** Add `--color-status-ok`, `--color-status-warn`, `--color-status-danger` (and their background/border variants `--color-status-danger-bg`, `--color-status-danger-border`) to `colors-and-type.css`. Replace all raw hex in admin.

### C3 — Admin has zero mobile responsive layout

**File:** `apps/admin/src/app/globals.css`  
**Risk:** The sidebar-plus-main grid is hardcoded `grid-template-columns: var(--sidebar-w, 240px) 1fr`. Below about 900px viewport width the content column loses all usable space. On a phone or iPad the topbar overflows.  
**Severity:** Medium-high — the admin is owner-only so blast radius is small, but if the owner ever checks it on an iPad the entire UI is unusable. A single `@media (max-width: 768px)` that collapses the sidebar to a full-width drawer or an icon-only strip would resolve it.

## High-priority findings

### H1 — `box-shadow` used in two typewars game states (brand: borders only, no shadows)

**File:** `apps/typewars/src/styles/typewars.css` lines 51, 281  
```css
.legion-card.picked { border-color: var(--lc); box-shadow: inset 0 0 0 1px var(--lc); }
.word-card.matching { border-color: var(--legion-color); box-shadow: inset 0 0 0 1px var(--legion-color); }
```
These use `box-shadow: inset 0 0 0 1px` to simulate a double-border on selected states. The intent is correct (showing pick/active state with legion colour), but it violates the brand prohibition on shadows. The same visual effect is achievable with `outline: 1px solid var(--lc); outline-offset: -2px;` (which survives border-radius on modern browsers) or CSS custom property `border-width` promotion.

### H2 — `font-weight: 600` in three typewars topbars (brand: 400/500 only)

**Files:** `apps/typewars/src/components/LegionSelect.tsx:21`, `apps/typewars/src/components/MapWarMap.tsx:104`, `apps/typewars/src/components/Leaderboard.tsx:70`  
```tsx
<span className="ss-mono" style={{ fontWeight: 600 }}>typewars.sastaspace.com</span>
```
The brand guide says "two font weights only: 400 regular, 500 medium. No 600+." The typewars domain name appears in the topbar of every screen and is the most-seen text. This needs to be `fontWeight: 500` (or `var(--fw-medium)`).

### H3 — `fontWeight: 600` in leaderboard "you are me" row

**File:** `apps/typewars/src/components/Leaderboard.tsx:145`  
```tsx
<span style={{ display: 'flex', alignItems: 'center', gap: 8, fontWeight: isMe ? 600 : 400 }}>
```
Same weight violation as H2. The distinction between "me" and "others" in the roster should use colour (e.g., the orange `lb-you-tag` chip already present) rather than a forbidden weight.

### H4 — ALL CAPS button labels in typewars (brand: sentence case only)

**Files:** `apps/typewars/src/components/LegionSelect.tsx:86`, `apps/typewars/src/components/LegionSwapModal.tsx:70`  
```tsx
{submitting ? 'enlisting…' : 'ENLIST →'}
// and
CONFIRM SWITCH →
```
The brand guide is explicit: "Sentence case only in UI. No ALL CAPS." These are the primary CTA buttons on two high-traffic screens. Change to `'Enlist →'` and `'Confirm switch →'`.

### H5 — ALL CAPS "YOU" tag in leaderboard

**File:** `apps/typewars/src/components/Leaderboard.tsx:148`  
```tsx
<span className="lb-you-tag">YOU</span>
```
Same brand violation. Change to `you` or `you →` (consistent with the mono-styled, lowercase aesthetic of the rest of the leaderboard).

### H6 — Focus styles are `border-color` changes only; no visible keyboard ring on interactive elements

No app has a `focus-visible` ring that would satisfy WCAG 2.4.7 (Focus Visible). The input `:focus` handler changes border colour (dark on paper — contrast is adequate), but interactive non-input elements (buttons, nav items, word cards, legion cards, topbar links, leaderboard rows) have no explicit focus-visible style. Keyboard navigation is functional but the focus position is invisible. This is an accessibility fail.

## Medium findings

### M1 — Six unlisted body-copy colours in landing and notes

`#3a3834` is used six times in `landing.module.css` and once in `notes.module.css` for descriptive lede/paragraph text. `#2b2a28` appears once in `landing.module.css` for the "about" column copy. Neither is a registered design token. The luminance of `#3a3834` on paper (`#f5f1e8`) is approximately 8.5:1, so WCAG AA body is met, but the value is not aliased. These should be `--brand-ink` (which is `#1a1917`, slightly darker) or a new `--brand-body-alt` token, whichever matches design intent.

### M2 — `color-surface` (#ffffff) used for form field backgrounds in notes/admin while page background is warm paper

**Files:** `apps/notes/src/components/auth.module.css:76`, `apps/notes/src/components/comments.module.css:142`, `apps/admin/src/app/globals.css` (multiple)  
Form inputs render on a pure-white (`#ffffff`) field, floating inside a warm-paper (`#f5f1e8`) page. The contrast between the field and the surrounding surface is noticeable — a micro-level jarring detail. The token `--color-surface: #ffffff` exists intentionally, but the visual effect may not be what the designer intended. Consider `--color-surface: var(--brand-paper-lifted)` (`#fbf8f0`) for inputs to keep the warm palette consistent.

### M3 — Two transition declarations missing `var(--ease-std)`

**File:** `apps/typewars/src/styles/typewars.css` lines 94, 180  
```css
.swap-pill { cursor: pointer; transition: border-color 120ms; }
.rql-row { … transition: background 120ms; … }
```
Both use a raw `120ms` duration with no easing curve — they will default to the browser's `ease` function. All other transitions in the codebase use `120ms var(--ease-std)` for consistency. Two-line fix.

### M4 — Typewars topbar shows hardcoded `season 1 · day 12 / 30` string in three components

**Files:** `LegionSelect.tsx:23`, `MapWarMap.tsx:105`, `Leaderboard.tsx:80`  
The season counter is hardcoded. If this is live data it should come from the server; if it is display-only metadata it should at least be a single shared constant in `/lib/`. Currently it is three independent string literals that can drift out of sync.

### M5 — Admin topbar uses `backdrop-filter: blur(8px)` but the background is `var(--color-bg)` (opaque)

**File:** `apps/admin/src/app/globals.css:258`  
```css
.topbar { background: var(--color-bg); … backdrop-filter: blur(8px); }
```
`backdrop-filter` on an opaque element does nothing visible. The property is not harmful but wastes a GPU composite layer and signals a copy-paste from the typewars topbar (which uses `rgba(245,241,232,0.85)`). Either make the topbar translucent or drop the property.

### M6 — Notes app `@theme` block in `globals.css` is Tailwind v4 but notes doesn't appear to use Tailwind utilities

`apps/notes/src/app/globals.css` imports `tailwindcss` and defines `@theme` variables, but the notes module CSS is pure CSS Modules with no Tailwind class names in any component. The `@import "tailwindcss"` injects a full reset and all utilities (significant bundle weight). If notes doesn't use Tailwind utilities, replacing `@import "tailwindcss"` with a minimal reset block would reduce the CSS payload.

### M7 — typewars layout.tsx does not specify font weights explicitly

**File:** `apps/typewars/src/app/layout.tsx:5`  
```ts
const inter = Inter({ subsets: ['latin'], variable: '--font-inter', display: 'swap' });
```
No `weight` array is specified. `next/font` defaults to loading all available weights (300, 400, 500, 600, 700, 800, 900 for Inter). Landing and notes explicitly pass `weight: ['400', '500']`, which reduces the font payload by ~70%. Typewars loads the full weight axis unnecessarily, which slows first paint.

## Recommended next 5 actions

**Action 1 — Extract status-colour tokens and replace all raw hex in admin (C2 + C3)**  
Add `--color-status-ok / -warn / -danger` (and their `-bg`, `-border` variants) to `colors-and-type.css`. Grep-replace across `admin/src/app/globals.css` and all `apps/admin/src/components/panels/*.tsx`. Estimated: 90 minutes. Impact: eliminates two-colour-drift reds today, makes future palette tweaks a one-file change.

**Action 2 — Wire admin to the shared design-token package (C1)**  
Add `@sastaspace/design-tokens` as a dev dependency to `apps/admin/package.json` and replace the duplicated `:root` block with `@import "@sastaspace/design-tokens/colors-and-type.css"`. Estimated: 20 minutes. Impact: admin stays in sync with every future token change for free.

**Action 3 — Add a global `focus-visible` ring (H6)**  
Add to each app's `globals.css` (or to `colors-and-type.css`):
```css
:focus-visible {
  outline: 2px solid var(--brand-sasta);
  outline-offset: 2px;
}
```
Then audit each app's interactive elements to ensure none suppress `outline` without providing an alternative. This alone advances the project from WCAG-missing to WCAG AA on focus. Estimated: 2 hours including testing.

**Action 4 — Replace `box-shadow: inset` with `outline` on typewars selected states (H1)**  
`.legion-card.picked` and `.word-card.matching` in `typewars.css`: replace `box-shadow: inset 0 0 0 1px var(--lc)` with `outline: 1px solid var(--lc); outline-offset: -2px;`. This enforces the brand rule and actually renders more crisply on HiDPI screens than a box-shadow. Test on Safari (outline-offset inside border-radius has a known rendering quirk; the fallback is to use `border-width` promotion with custom properties). Estimated: 30 minutes.

**Action 5 — Add an admin mobile layout (C3)**  
This is the only app with zero responsive handling. A minimal fix: below 900px, switch the `.app` grid to `grid-template-columns: 1fr` and render the sidebar as a slide-in drawer (already close — the `.app.collapsed` state does the grid transition). Alternatively, add a hamburger toggle at 768px. Estimated: 3 hours. Impact: makes the owner-facing admin usable on an iPad or remote mobile check.
